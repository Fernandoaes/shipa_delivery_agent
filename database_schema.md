# Shipa Inbound Agent — Database Schema

> PostgreSQL. Two groups of tables: **read side** (`customers`, `orders`) synced from **Twin** via the backend, and **write side** (everything else) created by the agent and dashboard. All primary keys are `uuid` (generated by us). Twin's own identifiers are stored as separate unique reference columns so we never collide with their keys.

---

## Read side — synced from Twin

The backend pulls from Twin and upserts these. They are effectively a local, queryable cache so the agent gets sub-second lookups and we have a stable order snapshot at call time.

### `customers`
One row per customer, keyed for caller-ID lookup. If Twin is order-centric and has no customer entity, derive this from the order feed (dedupe on phone).

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `customer_id` | uuid | **PK**, default gen | our internal id |
| `twin_customer_ref` | text | unique, nullable | Twin's id if it exists |
| `full_name` | text | not null | |
| `primary_phone` | text | not null, **unique**, **indexed** | E.164; main lookup key |
| `alt_phone` | text | nullable | |
| `language_pref` | text | nullable | `en` / `ar`, if known |
| `last_synced_at` | timestamptz | not null | |

Constraint: `uq_customers_primary_phone` unique on `primary_phone` (one customer per number).
Index: `idx_customers_primary_phone` on `primary_phone`.

## Relationships at a glance
- `customers` 1 ─ ∞ `orders` (a customer has many orders)
- `customers` 1 ─ ∞ `calls`, `orders` 1 ─ ∞ `calls`
- `calls` 1 ─ ∞ each operational table (`verifications`, `reschedules`, `investigations`, `escalations`, `address_flags`, `merchant_referrals`, `fallback_messages`). All but `verifications` and `fallback_messages` are constrained to **at most one row per call**.
- `orders` also 1 ─ ∞ `reschedules`, `investigations`, `address_flags`, `merchant_referrals` (so ops can view by order or by call)

## Twin sync notes
- Upsert `orders` on `twin_order_ref`; refresh `status`, `delivery_window`, `otp_code`, `assigned_driver`.
- Decide cadence: live pull on lookup (freshest, slower) vs. periodic sync + cache (faster, can be stale). For status and OTP I'd lean toward a **live pull at call start** so the agent never reads a stale code; everything else can be cached.
- `last_synced_at` lets the dashboard show data freshness and lets us detect a broken sync.

## Sensitive-data notes
- `otp_code` and PII (`primary_phone`, `delivery_address`, `full_name`): encrypt at rest, restrict in API responses, and scrub the OTP from stored transcripts where the platform allows.
- Confirm UAE data-residency requirements before choosing where Postgres lives.
